OpenAI 前几天发了一篇文章，叫 [Harness engineering: leveraging Codex in an agent-first world](https://openai.com/index/harness-engineering/)。

我读完以后，最有感触的地方不是“他们用 Codex 写了多少代码”。

真正重要的是另一件事：当 agent 开始承担越来越多软件生命周期工作，工程师的工作重心会往后退一层。你不再只是亲手写代码，而是要设计环境、表达意图、构建反馈回路，让 agent 能自己把事情做完。

这个变化，我愿意粗暴地概括成两句话。

Harness Engineering 解决的是：让 agent 能工作。

Standard Engine 解决的是：让 agent 按团队认可的标准工作。

前者是眼睛、手和反馈回路。

后者是工程判断。

OpenAI 那篇文章里讲了很多具体做法：让 Codex 能启动应用，能操作浏览器，能看 UI、日志、metrics、trace，能复现 bug，能验证修复，甚至能在一条任务里跑几个小时，自己循环推进。

这些东西很关键。

但我一直觉得，harness 化跑起来以后，很快会撞到下一个问题：

agent 已经能跑，能改，能测，也能交付证据。

可它到底按什么标准判断自己做对了？

这个问题如果不解决，AI Coding 会从“写不动”变成另一种麻烦：写得太快，改得太勤，坏模式扩散得太猛。

## Harness 的本质不是测试

很多人一看到 harness，就会想到测试脚本、CI 命令、浏览器自动化、回归用例。

这当然没错。

只是太窄了。

Harness 真正有价值的地方，在于给 agent 一个可行动的环境。测试只是其中一部分。

它至少要解决四件事。

第一，agent 能启动系统。

它不能只在静态代码里猜。它要能把应用跑起来，能进到真实页面、真实服务、真实任务里。

第二，agent 能观察系统。

页面长什么样，日志怎么报，网络请求有没有错，trace 里卡在哪里，任务状态有没有推进，这些东西不能都靠人截图喂给它。

第三，agent 能复现问题。

一个 bug 不再只是一段自然语言描述，而要变成一条可以反复执行的路径。

第四，agent 能验证修复。

它不能改完代码就宣布结束。它要重新跑关键路径，留下证据，说明这次修复到底覆盖了什么。

所以 harness 的核心作用，是把软件工程从“人盯着看”往前推到“agent 可感知、可操作、可反馈”。

它降低的是人类确认成本。

这个方向非常对。

但它还不够。

因为 harness 通常能告诉 agent：这个检查失败了。

它不一定能告诉 agent：这个失败为什么重要，违反了哪条工程判断，正确的修复方向是什么，同类问题以后该不该自动拦住。

很多判断还停在人脑里。

停在 review comment 里。

停在聊天记录、事故复盘、口头约定里。

这些东西不沉淀，agent 会一遍遍踩同样的坑。模型更强以后，它只是踩得更快。

## 新瓶颈：团队判断

AI Coding 的早期瓶颈是生成能力。

模型能不能写出代码？

能不能理解需求？

能不能补测试？

能不能别把项目搞挂？

进入 harness 化以后，瓶颈会移动。

agent 可以更快地写代码，也可以更快地跑验证。此时真正稀缺的东西变成了团队判断。

什么样的代码算可维护？

哪些边界不能被突破？

什么样的测试证据才够？

哪些临时绕路会制造长期债务？

哪些输出虽然能跑，但不应该被接受？

这些判断，过去靠资深工程师 review。

可问题来了：agent 的吞吐量远高于人工 review 的吞吐量。

如果每次都靠人重新解释一遍，标准就规模化不了。

还有一个更隐蔽的问题。

agent 很擅长复制既有模式。

仓库里有好模式，它会复制好模式。仓库里有坏模式，它也会复制坏模式。

当 AI Coding 的速度上来以后，坏模式的扩散速度也会跟着上来。以前一个坏写法可能几周才蔓延开，现在一天就能长得到处都是。

继续加强 harness 是必要的。

只是光靠 harness 不够。

我们还需要把团队反复做出的工程判断，转成 agent 可以识别、引用、执行和修复的标准系统。

我把这件事叫 Standard Engine。

## Standard Engine 是什么

Standard Engine 不是某个具体工具。

也不是一份更长的 AGENTS.md。

它是一套工程系统：把团队长期有效的判断，转化成仓库内可命名、可版本化、可验证、可追溯、可修订的标准对象。

它站在 harness 之上。

Harness 提供运行、观察和反馈。

Standard Engine 给这些反馈补上判断语义。

可以把 AI Coding 的工程系统分成三层。

Prompt 层：告诉 agent 要做什么。

Harness 层：让 agent 能运行、观察、测试、复现、提交证据。

Standard Engine 层：告诉 agent 什么才算做对，失败属于哪个标准，应该往哪里修。

更短一点：

Harness 是 agent 的感知和行动回路。

Standard Engine 是 agent 的工程判断回路。

没有 harness，agent 容易在代码里猜。

没有 Standard Engine，agent 容易为了通过检查，绕过真正的问题。

## 它不能写成一本厚规范

标准工程化最容易走错的一条路，是写一份越来越长的“AI 开发规范”。

这通常没什么用。

一份巨大的规则文档会很快腐烂。它会占上下文。它会把重要信息和次要信息混在一起。它还会让 agent 不知道优先级。

最后它变成一种心理安慰：

人觉得自己写了规范。

系统并没有真正执行规范。

Standard Engine 应该反过来做。

入口要短。

标准要分散。

判断要命名。

检查要机器可执行。

报告要能被 agent 消费。

每个标准都要有清楚边界。别把所有“最佳实践”塞进同一份文档里。

一个最小的标准对象，我觉得至少要包含这些字段：

- id：稳定编号，用在报告、日志、review 和交付记录里。
- intent：这个标准保护什么工程判断。
- scope：适用于哪些代码、流程或产物。
- enforcement：由什么检查、测试、harness 或 review 机制执行。
- evidence：什么证据能证明它被满足。
- repair path：失败后应该怎么修，不能只写“禁止”。
- revision trigger：什么情况下需要更新标准本身。

这几个字段看起来普通，其实很要命。

它们把“资深工程师脑子里的判断”，变成了 agent 可以读取、引用、执行的对象。

标准一旦有了 id，后面的事情才连得起来。

检查失败可以带标准 id。

运行证据可以引用标准 id。

PR 描述可以说明本次满足了哪些标准。

事故复盘可以反向更新标准。

质量趋势也能按标准聚合，而不是散成一堆零碎报错。

## 它也不是普通 CI

CI 通常回答一个问题：

过了没？

Standard Engine 要回答的问题更多。

违反了哪个标准？

证据是什么？

这个标准为什么存在？

应该往哪个方向修？

这次失败是实现问题，还是标准本身该修了？

同类问题以后能不能自动发现？

比如，一个普通 CI 可能只会说：某个文件超过行数限制。

Standard Engine 不应该停在这里。

它应该继续说明：这次失败违反的是“复杂度边界”标准；这个标准保护的是模块可读性和 agent 可导航性；证据是当前文件行数、阈值、历史豁免状态；建议修复方向是按服务、组件、hook、fixture 或领域边界拆分；如果这个限制反复阻碍合理演进，再触发标准修订。

这就不是简单检查了。

这是把工程判断注入反馈回路。

对人来说，这是更好的错误信息。

对 agent 来说，这是更明确的修复上下文。

## 把 taste 工程化

OpenAI 那篇文章里有一个很重的点：架构和 taste 需要被执行，不能只靠描述。

这句话放到 AI Coding 里特别狠。

在人类团队里，taste 很多时候是靠长期协作传递的。

什么样的抽象太早。

什么样的文件太大。

什么样的依赖方向危险。

什么样的日志足够定位问题。

什么样的 UI 状态才算可验证。

这些东西常常没完整写在规范里。新人靠 review 慢慢学，老人靠直觉当场拦。

agent 没有这个过程。

它只继承两样东西：可见上下文，以及仓库里已经存在的模式。

所以 Standard Engine 的一个核心价值，就是把 taste 拆成可执行标准。

一开始不需要很宏大。

从高频问题开始就行。

代码结构有没有超过可读边界。

接口层有没有塞进太多业务逻辑。

数据访问有没有绕开统一入口。

旧的复杂模块有没有继续变胖。

文档和 agent 入口有没有保持同步。

功能交付有没有留下足够验证证据。

关键产物有没有达到可用质量门槛。

这些都不只是格式问题。

它们是团队对长期可维护性的判断。

过去靠 review 反复提醒。现在要把它们变成系统能力。

## 对抗 agent 规模化后的熵增

AI Coding 最大的风险之一，不是 agent 写不出代码。

是 agent 写得太快。

速度本身没问题。

问题是，如果系统边界不够强，速度会放大熵增。

一个坏模式，在人工团队里可能要几周才扩散开。agent 高吞吐以后，可能一天内复制到很多地方。

一个临时 workaround，如果没有被标记和清理，很快就会变成“仓库惯例”。

这就是我很在意 Standard Engine 的原因。

Harness 能发现一部分问题，但它更偏运行反馈。Standard Engine 要负责把重复出现的问题提升为标准，再交给检查、报告和修复路径持续执行。

这有点像工程里的垃圾回收。

人工 review 里反复出现的问题，不该永远停在 comment 里。

线上事故暴露的系统性缺陷，不该只停在复盘文档里。

某次 agent 绕路成功但长期有害，也不能只靠下次人类记得提醒。

它们应该被提升为标准。

一旦成为标准，后面的 agent 再遇到同类问题时，不只是被拦住。

它还会知道自己为什么被拦住，应该怎么修，修完要留下什么证据。

## Standard Engine 怎么运转

一个能落地的 Standard Engine，不需要一开始搞得很复杂。

先做四个动作就够了。

第一，检查输出携带标准 ID。

不要只输出“失败”，要输出“违反了哪个标准”。标准 ID 是追踪、统计和修复的共同语言。

第二，运行证据携带标准引用。

浏览器验证、回归脚本、质量报告、运行摘要，不只是保存结果，还要说明这些结果覆盖了哪些标准。

第三，交付记录绑定验证结果。

每次有意义的 agent 变更，都应该留下 prompt、目标、改动、验证、风险和提交关系。

不然未来的 agent 或 reviewer 很难理解当时为什么这么做。

第四，review 和事故反向更新标准。

如果一个问题反复出现，不要只修当前代码。要问几个更烦、也更有价值的问题：

是不是缺了一个标准？

是不是已有标准没有自动执行？

是不是错误信息没给修复方向？

是不是证据不足以复现判断？

这四个动作连起来，闭环就开始出现了。

标准定义判断。

Harness 提供证据。

检查发现偏差。

交付记录保存上下文。

Review 和事故推动标准进化。

## 工程负责人真正要管的东西变了

AI Coding 之前，工程负责人主要管人的协作效率、代码质量和交付节奏。

AI Coding 之后，这些当然还重要。

只是控制点变了。

你不能只靠增加 review 来管理 agent 吞吐量。

也不能只靠更好的 prompt 来获得长期一致性。

你要管理的是环境、反馈和标准。

环境决定 agent 能不能工作。

反馈决定 agent 能不能修正。

标准决定 agent 会朝哪个方向修正。

Harness Engineering 已经把环境和反馈往前推了一大步。

Standard Engine 要补上标准这一层。

它不是为了限制 agent。

恰恰相反，它是为了释放 agent。

边界、证据和修复方向足够清楚，agent 才能在更大范围里自治。人不用盯着每个细节，只需要管理系统如何吸收新的判断。

一个成熟的 AI Coding 团队，不会只问“模型能力够不够”。

它会问：

我们的工程判断有没有进入仓库？

我们的标准能不能被 agent 发现？

失败时有没有可执行的修复方向？

验证证据能不能复用？

review 经验是在沉淀，还是在流失？

## 结语

Harness 让 AI Coding 可运行。

Standard Engine 让 AI Coding 可治理、可继承、可持续扩张。

前者解决 agent 的手和眼睛。

后者解决 agent 的工程判断。

当 AI Coding 只是辅助个人写代码时，prompt 技巧可能已经够用。

当 AI Coding 进入团队级、仓库级、长期演进级的软件工程时，harness 是基础设施，Standard Engine 是治理系统。

未来优秀工程团队的差异，可能不只是用了哪个模型、接了多少工具、跑了多少测试。

更大的差异在这里：

有没有能力把自己的工程判断沉淀成可执行的标准系统。

这才是 AI Coding 从“能写代码”走向“能长期维护复杂系统”的关键一步。

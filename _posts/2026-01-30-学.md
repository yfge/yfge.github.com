# vibe‑coding 九阳神功之**学**：先问架构拿术语，再用术语组装新项目（会组合技术栈、会裁决）

你说的这种“学”，我觉得才是 vibe‑coding 的正路：

> 先拿一个成熟工程（例如 AI‑Shifu / Orion）当老师  
> → 问 AI：“这个系统的架构是什么？”  
> → AI 会吐出一堆术语（前后端分离、反向代理、数据库迁移、任务队列……）  
> → 你不用先把原理全学会，只要学会**怎么把术语组合成一个能交付的最小系统**  
> → 过程中再借助 AI 逐个“学术语”，但每个术语都必须能落到“下一步怎么做 + 怎么验收”

这就是 **学** 的定义：**会组合技术栈、会裁决，不懂原理也能不翻车。**

下面给你一套可重复的流程 + 一套可复制的提问模板，你照着问就能从 0 到 1 拼出专业系统。

---

## 0）准备：你要有一个“老师工程”和一个“练功工程”

老师工程（参考系统）：选一个就行

- `ai-shifu/ai-shifu`：Docker/Compose 工程化非常成熟
- `yfge/orion`：全栈骨架清晰（前端/后端/Nginx/Compose）

练功工程（你的新项目）：例如 `vibe-starter`  
先别追求完美，目标是：能跑起来、能验收、能迭代。

如果你的 AI 支持读本地仓库/文件，就把本地路径告诉它；  
如果不支持，就把 README + `docker/` 目录说明 + `docker-compose*.yml` 摘要贴给它（不用贴全仓库）。

---

## 1）第一问：先问“架构是什么？”把术语清单拿到手

你问 AI 的关键不是“讲课”，而是让它**输出结构化结果**。复制这段：

```text
请你分析这个项目的架构（以 ai-shifu 为例，也可以换成 orion）：

目标：
1) 用一页纸说明：它解决什么问题？核心模块是什么？请求/数据大致怎么流转？
2) 给出“架构术语清单”（20~40 个即可），按类别分组：
   - 入口与路由（例如 Nginx / 反向代理 / 前后端分离）
   - 前端（例如 Next.js / SSR / 静态资源）
   - 后端（例如 FastAPI / API 设计 / 鉴权）
   - 数据（例如 MySQL/Postgres / 连接池 / 迁移）
   - 异步（例如 任务队列 / 定时任务 / 重试）
   - 配置与交付（例如 .env / docker compose / dev 模式）
   - 可观测与运维（例如 healthcheck / 日志 / 监控）

输出要求（重要）：
- 每个术语只要“一句话解释 + 它解决的痛点”
- 最后给我一个“最小可复刻版本”（MVP）：如果我要用这些术语做一个新项目，哪些是必选？哪些可选？按优先级排序。
```

你拿到术语清单后，不要立刻全部上。小白最容易翻车就是“术语收集癖”：看着很懂，项目起不来。

---

## 2）第二问：从术语清单里“选配方”（组合技术栈）

你接下来要做的不是学术语，而是选一个能跑的组合。  
对小白最稳的默认配方通常是：

- 前端：Next.js
- 后端：Python / FastAPI
- 入口：Nginx（统一入口端口）
- 部署：Docker Compose（一键起全套）

你可以让 AI 用术语清单帮你做“选配方 + 砍功能”，复制这段：

```text
我现在要做一个新项目（vibe-starter），请你从你刚才给的“架构术语清单”里，选出一个最小可交付组合。

约束：
- 小白友好：越少组件越好
- 一键启动：docker compose up -d 就能跑
- 对外只暴露一个端口（例如 8080）
- 必须有最小验收：/healthz + /api/hello + 首页显示 “it works”

请输出：
1) 这次选用的“术语子集”（<=10 个），每个一句话解释“为什么需要它”
2) 新项目的最小架构图（用文字也行）
3) 服务清单（frontend/backend/nginx/数据库等）以及它们的职责
4) 一份按顺序的任务清单（每条任务都给验收命令）
```

这一步就是你说的“会合理组合技术”：  
不是把所有名词都堆上去，而是**拿最少的名词，拼出能交付的系统**。

---

## 3）第三问：学术语，但只学“够用版”（不翻车学习法）

拿到“术语子集”后，你就逐个问，但每个术语都只学三件事：

1) **它在新项目里要负责什么**（一句话）  
2) **最小怎么做**（只做 MVP）  
3) **怎么验收**（命令/URL/日志）

复制这个模板（每次替换术语即可）：

```text
我在做一个 Next.js + FastAPI + Nginx + Docker Compose 的最小系统。

术语：<填一个，例如 前后端分离 / 反向代理 / 数据库迁移 / 任务队列>

请你只回答这 6 件事（不要长篇讲原理）：
1) 一句话解释（<=30字，小白能懂）
2) 在我的系统里它负责什么？（<=2条）
3) 最小实现需要哪些“改动点”（用人话列清单：加哪个服务/加哪个配置/加哪个接口/加哪个脚本）
4) 最小验收怎么做？（docker compose / curl / 浏览器 URL）
5) 最常见的 3 个坑是什么？怎么避免？
6) 如果我现在不做它，会有什么后果？（帮我判断它是不是“必须”）
```

你会发现：你并不是在“学名词”，你是在**把名词变成可执行的工程动作**。

---

## 4）裁决：你不懂原理也能不翻车的 6 条底线

你说的“会裁决”，我建议你记这 6 条底线（小白阶段非常管用）：

1) **组件越少越好**：能不用队列就先不用；能不用微服务就别上  
2) **入口统一**：对外就一个端口（Nginx 统一入口），别让用户记一堆地址  
3) **先跑通再优化**：先把 /healthz + /api/hello + 首页跑通，再谈高级功能  
4) **数据库变更要留脚印**：只要你用了数据库，就尽早上迁移（哪怕最简）  
5) **异步后置**：队列/消息/重试是第二阶段（除非业务逼你）  
6) **每一步都要可验收**：AI 说“应该可以”不算，必须给命令/证据

当你纠结 A 还是 B 时，别凭感觉，直接用这个“裁决提问”：

```text
我在 A/B 两个方案之间纠结：
- A: <方案A>
- B: <方案B>

我的约束：小白友好、可交付、可回滚、可验收、docker compose 一键启动。

请你输出：
1) 推荐结论（先给结论）
2) 为什么（用 5 条以内理由）
3) 如果选错会怎么翻车？（最可能的 3 种失败）
4) 最小回滚方案是什么？
```

---

## 5）把术语落到“新项目交付”：你只要盯住两个输出

你之前觉得“落地卡/验收卡花里胡哨”，我建议换个说法你就不抗拒了：

- **改动清单**：我到底要加什么/改什么？（防止 AI 发散、也防止你漏步骤）
- **验收命令**：我怎么证明它真的生效？（防止“讲得很懂但跑不起来”）

你要求 AI 每次都按这个顺序输出即可：

```text
请先输出改动清单（按路径列出更好，但不是必须），我确认后你再逐文件生成内容。
每个改动都必须给验收命令。
```

这不是形式主义，这是小白最省心的护栏：**你不用懂很多，但你永远知道下一步做什么、怎么判断做对了。**

---

## 6）你现在就能开始练：用 AI‑Shifu 问架构 → 选 10 个术语 → 拼一个新项目

最推荐的练法（3 次循环）：

1) 问 AI‑Shifu 架构 → 拿术语清单 → 选 <=10 个术语 → 拼一个最小系统  
2) 再问 Orion 架构 → 对照它的术语清单 → 你会更懂“哪些术语是必选、哪些是装饰”  
3) 把两个清单的交集当成你的“默认配方库”（以后做项目就从这套配方开始）

做到这一步，你就真正掌握了 vibe‑coding 的“学”：  
**会问架构、会拿术语、会组合技术、会做裁决，最后能交付一个跑得起来的系统。**

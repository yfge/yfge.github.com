用 Vibe Coding 做了七八个项目上线的人来说几句实话。

先亮立场：**支持。但大部分人对 Vibe Coding 的理解是错的。**

我看了一圈回答，发现争论的焦点基本上是"AI 写代码靠不靠谱"。但这根本不是 Vibe Coding 的核心问题。Karpathy 原话说的是"fully giving in to the vibes"——完全跟着感觉走，不去读 AI 写的代码。

说实话，我一开始也觉得这是扯淡。一个写了十几年代码的人，突然让我不看代码？那和闭着眼过马路有什么区别？

然后我真的试了。结论是：**Karpathy 说的是一个极端情况，但他指向的方向是对的。**

---

## 我理解错了什么

去年做 Orion（一个统一通知网关，开源的）的时候，我还是老派做法——AI 写代码，我一行一行审。后端 FastAPI + SQLAlchemy，前端 Next.js + Tailwind，Docker Compose 部署。整个过程我像个质检员一样盯着每一行。

效率提升了大概 3 倍，但体验很累。我花了大量时间在"确认 AI 写得对不对"上面，而不是在"思考这个功能该不该这么做"上面。

转折点是做 agentskill.work 的时候。那天下午我有个想法：做一个 GitHub 上 Agent Skill 相关仓库的目录站。我试着放开一点，不再逐行审代码，而是只盯关键节点。

半天上线了。

从 GitHub API 抓取到 Celery 定时任务，从 Nginx 反代到 Let's Encrypt 证书，从 Docker Compose 编排到 Umami 埋点——AI 全写的，我没审过大部分代码。

那它炸了吗？

炸过。GitHub API rate limit 把我打死了一次。Nginx 配置有个 proxy_pass 多写了个斜杠导致 404。证书续期的 cron 第一版跑在容器里、容器重建就丢了。

但这些坑，**不是靠审代码能发现的**。这些是系统层面的坑——你得知道 GitHub API 有 rate limit 这回事，你得知道 Nginx 的 proxy_pass 尾部斜杠行为，你得知道 Let's Encrypt 的证书数据必须持久化。

这些是经验，不是代码审查。

那一刻我悟了：**Vibe Coding 不是"不看代码"，是"把注意力从代码层拉到系统层"。**

---

## 所以"规"比"看"重要

我后来做了一套工作流（写了个系列叫"九阳神功"，有兴趣可以去翻我博客），核心就是一件事：**别审代码，审上下文。**

具体来说，我写了一个叫 CLAUDE.md 的文件放在项目根目录。内容不复杂，就是告诉 AI：项目用什么技术栈、目录结构长什么样、提交规范是什么、哪些东西不能碰。

然后每次开工之前，先让 AI 生成一份 tasks.md——拆成 3-5 个 Phase，每个 Phase 独立可验收，每个任务精确到文件级别。做完一个打勾、commit 一次。

这套东西的效果是：我不需要看 AI 写的代码到底对不对，我只需要验收结果对不对。

```bash
# 验收后端
curl http://localhost:8080/api/hello
# 返回 200 带数据？过。
# 返回 500？回退，让 AI 重写这一个任务。
```

我有次做 fh_java 项目的重构，14 个 Controller 要改。用 tasks.md 一个一个推，每个 Controller 一个 commit，总共 14 个原子提交。中间有两次 AI 改错了——一次是改了认证逻辑导致别的接口 401，一次是把分页参数名改了但前端没同步更新。两次都是 `git revert` 回去重来，不到五分钟搞定。如果没有原子提交、14 个 Controller 改动搅在一起，我大概率直接 `git reset --hard` 全部重来。

你看，这个过程里我根本没有"审代码"。我审的是任务粒度、验收标准、和 commit 的干净度。

## 反对 Vibe Coding 的人，其实在担心什么？

我观察到反对者的核心焦虑是：**你不看代码，出了 bug 怎么办？**

这个担心完全合理。但我想反问一个问题：你看了代码，就能发现所有 bug 吗？

我写了十几年代码，code review 做过无数次。说句不好听的，大部分 code review 发现的问题是格式、命名、和"这个变量可以用 final"——真正的逻辑 bug、并发问题、边界条件，code review 的发现率并不高。

能发现严重问题的，是测试、是监控、是线上流量打过之后的报警。

所以我的做法不是"信任 AI 的代码"，而是"**给 AI 加刹车系统**"：

- Git 是第一道刹车。每做一步就 commit，炸了能回退。
- tasks.md 是第二道刹车。AI 只做清单上的事，不许自由发挥。我有一条写死的 prompt："不要修改当前 Phase 以外的任何文件。" 百分百有效吗？不是，隔几次它还会偷偷加私活——但至少比没有约束好太多了。
- 验收驱动是第三道刹车。curl 打一下、浏览器开一下、docker compose ps 看一下——结果对就行，代码怎么写的我不管。

有了这三道刹车，我对 AI 代码的信任度从 30% 提到了 80%。剩下 20% 靠什么？靠上线后的监控和告警。跟人写的代码一样——你也没法 100% 信任自己写的代码啊。

## 一个例子说明 Vibe Coding 的真实状态

前阵子我想把博客文章发到 X.com 的编辑器里，保留格式。这事听着特别简单，复制粘贴不就行了？

AI 第一版：纯文本输出。粘贴过去格式全丢。

我说要富文本。AI 改用 `pbcopy`。还是纯文本——`pbcopy` 只支持纯文本，这个我知道但 AI 不会主动告诉我。

我说用 `NSPasteboard`。AI 建了虚拟环境搞定了，但代码块没了。

我说 X.com 不支持代码块。AI 转引用块。表格又没了。

我说也不支持表格。AI 转列表。终于能用了。

40 分钟，来来回回五六轮。

这就是 Vibe Coding 的真实体验：**AI 像一个手速极快但缺乏常识的实习生。** 你说往东它马上跑，不说它就一路走到黑。

但换个角度想——如果没有 AI，这个需求我怎么解决？去 GitHub 上找现成工具？clone 下来、读代码、理解结构、配环境、跑起来——大概率比 40 分钟更久。而且别人的工具是通用的，不知道我的 `_posts` 目录在哪、不知道我的文章格式是什么。

AI 写的东西虽然改了五六轮，但每一版都完全贴合我的项目。这是"现场定制"，不是"通用工具"。用完即走。

## 说点可能得罪人的话

我觉得现在反对 Vibe Coding 的人分两种：

一种是真的在大型系统上吃过亏的工程师，担心代码质量失控。这种担心合理，但解决方案不是"回到手写代码"，是"建更好的工程约束"——我前面说的 CLAUDE.md、tasks.md、原子提交、验收驱动，就是在做这件事。

另一种是……怎么说呢，把"能手写代码"当作护城河的人。代码是他们的安全感来源。AI 一来，这个安全感没了，所以本能地抵触。

我理解这种心态，因为我自己经历过。去年某天用 Claude Code 半小时搞定了一个以前要写两天的功能，我第一反应不是开心，是恐慌——"卧槽这也行？那我十几年的经验算什么？"

后来我想明白了：**十几年的经验不是"能写代码"，是"知道该写什么代码"。** AI 把"写"的成本打到接近零了，但"判断"的成本一分没少。

你知道什么时候该用 bcrypt 而不是 MD5。你知道分页接口不能不加 limit。你知道 Docker Compose 的 depends_on 不等于"等你启动完我再启"。你知道 Nginx 的 proxy_pass 尾部加不加斜杠效果完全不同。

这些判断力，AI 不会替你做。

所以 Vibe Coding 的"Vibe"，真正指的不是"跟着感觉写代码"——而是"**跟着经验做判断，写代码的事交给 AI**"。

经验越深的人，Vibe Coding 越好使。这是个反直觉的结论。你以为 Vibe Coding 是给小白玩的？恰恰相反——小白 Vibe Coding 写出来的东西上不了线。真正能 Vibe 起来的，是那些知道系统会在哪里炸的老手。

## 回到问题

支持 Vibe Coding 吗？

**支持。但前提是你得有足够的工程经验来给 AI 装刹车。**

没经验的人玩 Vibe Coding，就像让驾校学员开赛车——车确实快，但第一个弯就飞出去了。

有经验的人玩 Vibe Coding，才是真正的效率倍增器。AI 给了速度，你补上方向感和刹车。

所以与其争论"支不支持 Vibe Coding"，不如问自己一个更实际的问题：**你有没有足够的经验，让 AI 替你写代码而不翻车？**

如果有——恭喜，这是你的时代。

如果没有——别急着 Vibe，先老老实实补课。技术判断力不是靠 Vibe 出来的，是靠被线上事故教育出来的。
